# SOCForge — SOC Analyst II Training Simulator

A full-stack, self-hosted SOC training range that looks and works like a modern SIEM — dark dashboard, field-based search language, live alerts, case management, and guided investigations with scoring.

**Every few minutes the alert engine injects a simulated attack into the event store and raises an alert. Each alert becomes an interactive training case: objective, analyst checklist, optional hints, evidence collection, disposition, automatic scoring, and a generated incident report with the full explanation of the attack.**

> ⚠️ **Defensive training only.** All telemetry is synthetic, generated by templates. There is no malware, no exploit code, no credential theft, and nothing here interacts with real systems. Vendor-neutral: no Splunk software, branding, or assets are used — only a familiar SIEM-style workflow.

## Quick Start

### Step 0 — Download the project

Either clone with git:

```bash
git clone https://github.com/raynacamp0503-svg/socforge.git
cd socforge
```

…or, with no git installed: click the green **Code** button at the top of this page → **Download ZIP** → extract it anywhere → open a terminal in the extracted folder.

### Option A — Docker (easiest: one command)

Requires [Docker Desktop](https://www.docker.com/products/docker-desktop/) (free).

```bash
docker compose up --build
# then open http://localhost:8080
```

The SQLite database persists in the `socdata` volume. `docker compose down -v` resets everything.

### Option B — Run with Node.js

Requires [Node.js](https://nodejs.org) **22.13 or newer** (uses the built-in `node:sqlite` module — check with `node --version`).

**Windows shortcut:** after installing Node, just double-click **`start-socforge.cmd`** in the project folder. It starts both servers and opens your browser. Done.

Manually (any OS):

```bash
# Terminal 1 — API on :4000
cd server
npm install
npm run dev

# Terminal 2 — UI on :5173 (proxies /api to :4000)
cd web
npm install
npm run dev
```

Open http://localhost:5173. On first run the server backfills 24 hours of noise events and seeds 4 incidents (one pre-completed as an example report). The first **live** alert fires ~25 seconds after startup, then every 2–5 minutes (tune with `ALERT_MIN_SEC` / `ALERT_MAX_SEC`).

**Pausing:** the ⏸ button in the sidebar pauses live alert generation (and saved-detection firing) without stopping the app — background noise keeps flowing so searches stay realistic. Nothing runs when the server is stopped: alerts never accumulate while the app is off, and all your cases and scores persist in the database between sessions.

## Training Coach (beginner mode)

New to SOC work? Flip on the **🎓 Training Coach** in the sidebar. It's a 19-step interactive walkthrough that follows you across every screen and explains the concepts as you go — what a SIEM is, events vs. alerts, severity and SLA-driven triage, the search language, pivoting, evidence collection, the four dispositions, MITRE ATT&CK, and detection engineering. It offers a "Take me there" button when a step covers a different page, remembers where you left off (saved in your browser), and can be toggled on/off at any time. It ends by assigning you your first solo investigation.

## The Workflow (how to train)

1. **Watch the queue.** A toast pops when a live alert fires. Open the case.
2. **Read the objective and detection logic.** Click *Run in Search ↗* to see the raw events behind the alert.
3. **Work the checklist.** Pivot on users/hosts/IPs (every value in a results table is a pivot link).
4. **Attach evidence** — key events from Related Logs, plus your own notes.
5. **Write analyst notes**, choose a disposition (True Positive / False Positive / Benign / Needs Escalation), and **Submit & Score**.
6. **Read the feedback**: score breakdown, the correct disposition, and a full explanation of what the attack was and how a Tier 2 analyst should reason about it.

**Scoring rubric:** checklist 40 pts · evidence 20 pts · correct disposition 40 pts · **−10 per hint**. SLA timers per severity: critical 30 min, high 1 h, medium 4 h, low 8 h.

## Search Language Cheatsheet

```
index=windows EventCode=4625          failed logons
sourcetype=sysmon process_name=powershell.exe
severity=high user=*                  any user, high-severity events
index=network dest_ip=147.78.47.93    connections to an IP
src_ip=10.20.*                        wildcards anywhere in a value
index=proxy POST                      bare words = free-text match
```

Supported fields: `index`, `sourcetype`, `host`, `user`, `src_ip`, `dest_ip`, `EventCode`/`event_code`, `process_name`, `severity`, `scenario`. `*` matches anything; unknown fields fall back to matching the raw event JSON. Combine any number of terms (implicit AND) and pick a time range from the picker.

**Save as detection:** any query can be saved as a detection rule — it then runs against live events every 60 seconds and raises its own alerts (with a 10-minute cooldown). This is how you practice detection engineering and tuning.

## Scenario Builder

Add your own attack: drop a JSON file into `server/scenarios/` and restart the API. Full schema template is on the Scenarios page in the app. Highlights:

- `events[]` — templated log events with `offsetSec` (relative to alert time), `repeat`, `intervalSec`
- Placeholders (`{user}`, `{host}`, `{attacker_ip}`, `{c2_ip}`, `{dga_domain}`, `{cloud_domain}`, `{new_account}`, …) are filled with randomized entities every time the scenario fires, so no two cases look identical
- `checklist`, `hints`, `expectedEvidence`, `correctDisposition`, `explanation` drive the training mode and scoring
- `detection` is the SPL-style query shown as the alert's detection logic
- `reportTemplate` becomes the "Recommended Response Actions" section of the generated report

## Architecture

```
web/  (React + TS)                    server/  (Express)
┌─────────────────────┐   /api/*     ┌──────────────────────────────┐
│ Dashboard  Search   │ ───────────► │ routes.js   REST API         │
│ Queue      Case     │              │ search.js   SPL-like parser  │
│ Scenarios  Portfolio│ ◄─────────── │ engine.js   alert scheduler, │
└─────────────────────┘  SSE stream  │             scoring, reports │
                                     │ generator.js noise + seeding │
                                     │ scenarios/  JSON attack defs │
                                     │ db.js       SQLite (node:sqlite)
                                     └──────────────────────────────┘
```

- **Noise generator**: benign events (logons, process creation, DNS, firewall, proxy) every ~4 s, so real attacks hide inside realistic background traffic.
- **Alert engine**: picks a random scenario every 2–5 min, instantiates it with random entities, injects its events, creates the alert + case, and broadcasts over SSE.
- **Detection runner**: user-saved queries evaluated every 60 s against fresh events.
- **Retention**: noise events older than 7 days are pruned; alert-linked events are kept.

## Safety Statement

This project is strictly defensive training infrastructure:
- All "attacks" are pre-written **log templates** — text records that describe activity which never occurred.
- No malware, exploit code, payloads, or offensive tooling of any kind.
- Nothing scans, probes, authenticates to, or communicates with any system other than its own database.
- Indicators (IPs, domains) are fabricated or reserved-range values.
